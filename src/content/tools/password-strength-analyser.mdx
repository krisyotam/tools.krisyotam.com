I want a password strength analyzer that breaks down exactly why a password is strong or weak. For any given password, show: its length, which character classes it uses (lowercase, uppercase, digits, symbols), the size of the resulting character pool, the Shannon entropy in bits, and an estimated time to crack assuming 10 billion guesses per second (roughly what a modern GPU cluster achieves against fast hashes like SHA-256).

The entropy calculation is `length * log2(poolSize)` where pool size is the sum of character class sizes present in the password. If only lowercase letters are used the pool is 26; add uppercase and it becomes 52; add digits for 62; add symbols for roughly 95. This is an upper-bound estimate that assumes the attacker knows which character classes are used but must brute-force the actual characters. Real-world attacks use dictionaries, rules, and Markov chains, so the effective entropy of a dictionary word is much lower than its character-level entropy suggests.

The crack time estimate assumes average-case search (half the keyspace) at 10 billion guesses per second. This rate is realistic for offline attacks against unsalted SHA-256 hashes using a cluster of GPUs. Against bcrypt at cost 12, the rate drops to perhaps 50,000 guesses per second, which changes the numbers dramatically. The estimate here represents a worst-case scenario for the defender.

The visual strength meter maps entropy ranges to five levels: very weak (under 28 bits), weak (28-35), fair (36-59), strong (60-79), and very strong (80+). These thresholds are opinionated but grounded in the reality that anything under 40 bits is trivially crackable and anything over 80 bits is beyond practical offline attack.
